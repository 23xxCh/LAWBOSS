# 出海法盾 CrossGuard — 需求规格文档 (spec.md)

## 1. 产品定位

**出海法盾（CrossGuard）** 是跨境电商智能合规审查平台，帮助出海电商卖家检测产品描述是否符合目标市场的法规要求，降低因违规导致的下架、罚款、诉讼等风险。

### 目标用户

- 跨境电商卖家（Amazon、Shopee、Lazada 等平台）
- 电商运营团队（Listing 优化、产品上架）
- 合规/法务人员（产品合规审核）
- 电商服务商（代运营、翻译、本地化服务）

### 核心价值

- **降低合规风险** — 上架前自动检测违规内容，避免平台处罚
- **提升运营效率** — 替代人工逐条审查，秒级生成合规报告
- **降低专业门槛** — 无需深入理解各国法规，系统自动匹配规则
- **一键合规修改** — 自动生成合规版本，减少人工改写成本
- **数据飞轮** — 用户反馈驱动规则优化，越用越准

## 2. 功能需求

### 2.1 合规检测（核心功能）

#### 2.1.1 文本合规检测

**输入**：
- 产品描述文本（支持中英文）
- 产品类别（化妆品、电子产品、食品、玩具、纺织品、膳食补充剂）
- 目标市场（EU、US、SEA_SG、SEA_TH、SEA_MY）

**输出**：
- 合规检测报告，包含：
  - 风险评分（0-100）
  - 风险等级（高/中/低）
  - 违规项列表（每项含：违规类型、违规内容、法规依据、严重度、修改建议）
  - 合规版本（自动替换违规词汇后的产品描述）
  - 必需标签列表
  - 必需认证列表
  - 修改建议汇总

#### 2.1.2 图片合规检测

**输入**：产品图片（PNG/JPG/WEBP）

**流程**：OCR 文字提取 → 文本合规检测

#### 2.1.3 违规类型

| 违规类型 | 标识 | 说明 | 优先级 |
|----------|------|------|--------|
| 医疗宣称 | `medical_claim` | 化妆品/食品声称具有治疗、预防疾病功能 | P0 |
| 绝对化用语 | `absolute_term` | 使用"最好"、"100%"等绝对化表述 | P0 |
| 虚假广告 | `false_advertising` | 功效宣称无科学依据，如"7天见效" | P0 |
| 缺失标签 | `missing_label` | 缺少目标市场强制要求的标签信息 | P1 |
| 禁用成分 | `banned_ingredient` | 含有目标市场禁用的成分 | P1 |

#### 2.1.4 风险评分规则

- 医疗宣称违规：25 分/项
- 绝对化用语违规：15 分/项
- 虚假广告违规：15 分/项
- 缺失标签违规：10 分/项
- 禁用成分违规：30 分/项
- 存在医疗宣称额外 +20 分
- 违规项 >= 3 额外 +10 分
- 上限 100 分

| 风险等级 | 分数范围 | 含义 |
|----------|----------|------|
| 高风险 | >= 70 | 建议立即整改，可能面临下架/罚款 |
| 中风险 | >= 40 | 存在违规可能，建议修改 |
| 低风险 | < 40 | 基本合规，可进一步优化 |

### 2.2 两级检测流水线

| 级别 | 引擎 | 特点 |
|------|------|------|
| 一级：关键词初筛 | 5 个 BaseChecker 子类 | 快速、确定性、高召回率 |
| 二级：AI 语义复筛 | AISemanticChecker (LLM) | 深度理解上下文、低误报率、检测隐含违规 |

AI 语义检测支持 OpenAI / DeepSeek / Ollama 后端，检测 5 个维度：隐含医疗宣称、误导性功效宣称、不当对比、文化/法律语境违规、成分暗示违规。

### 2.3 市场与类别支持

#### 2.3.1 支持的市场

| 市场 | 标识 | 状态 |
|------|------|------|
| 欧盟 | `EU` | 已实现 |
| 美国 | `US` | 已实现 |
| 新加坡 | `SEA_SG` | 已实现 |
| 泰国 | `SEA_TH` | 已实现 |
| 马来西亚 | `SEA_MY` | 已实现 |

#### 2.3.2 各市场产品类别

**EU**：化妆品、电子产品、食品、玩具、纺织品

**US / SEA_SG / SEA_TH / SEA_MY**：化妆品、电子产品、食品、膳食补充剂

#### 2.3.3 各市场法规体系

**EU 市场**：
- 化妆品：EC No 1223/2009（化妆品法规）、EC No 655/2013（宣称法规）
- 电子产品：CE 认证指令、RoHS 指令、RED 指令
- 食品：EC No 178/2002（一般食品法）、EC No 1169/2011（食品信息法规）
- 玩具：Directive 2009/48/EC（玩具安全指令）
- 纺织品：Regulation (EU) No 1007/2011（纺织品标签法规）

**US 市场**：
- 化妆品：FD&C Act、FDA 21 CFR 700-740
- 电子产品：FCC 认证、UL 认证
- 食品：FD&C Act、21 CFR 101（食品标签）
- 膳食补充剂：DSHEA、21 CFR 111

**SEA 市场（新加坡/泰国/马来西亚）**：
- 化妆品：ASEAN Cosmetic Directive (ACD)、各国 NPRA/HSA/FDA 监管
- 清真认证要求（马来西亚 JAK Halal）

### 2.4 必需标签与认证查询

用户可查询特定市场+类别下的：
- 强制标签要求（如成分表、CE 标志等）
- 必需认证/备案（如 CPNP 备案、CPSR 报告等）

### 2.5 合规版本生成

系统自动将违规词汇替换为合规表述，生成合规版本的产品描述。采用上下文感知替换：只替换被标记为违规的词汇，避免过度替换。

### 2.6 批量检测

支持批量提交多个产品描述进行合规检测，生成批量检测报告。

### 2.7 检测历史

保存用户的检测历史记录，支持：
- 查看历史检测报告
- 按时间、风险等级、市场筛选
- 导出检测报告（PDF）

### 2.8 用户认证与权限

- JWT + bcrypt 认证系统
- RBAC 角色控制：admin / user / viewer
- 所有核心端点需认证访问
- 删除报告、触发巡检需管理员权限

### 2.9 电商平台对接

- Amazon SP-API / Shopee Open API 抽象对接层
- 自动拉取平台 Listing → 合规检测 → 高风险告警
- 合规巡检引擎：定时拉取 + 检测 + Webhook 通知

### 2.10 数据飞轮

用户反馈机制驱动规则持续优化：
- 误报标记（false_positive）→ 建议移除该词
- 漏报标记（false_negative）→ 建议补充词库
- 正确标记（correct）→ 增强规则置信度
- 自动生成规则优化建议
- 检测精度指标：准确率、误报率、漏报率

## 3. 非功能需求

### 3.1 性能

- 单次文本检测响应时间 < 2 秒
- 批量检测（100 条）响应时间 < 30 秒
- 并发支持 >= 50 QPS

### 3.2 准确性

- 医疗宣称检测召回率 >= 95%
- 绝对化用语检测召回率 >= 90%
- 误报率 < 10%

### 3.3 可用性

- 支持中英文界面
- 检测结果可视化展示（风险仪表盘、违规高亮、Diff 对比）
- 操作步骤 <= 3 步完成一次检测
- 每个违规项可一键标记误报/正确

### 3.4 安全性

- 用户数据加密存储
- API 接口认证鉴权（JWT Bearer Token）
- 产品描述数据不对外泄露
- CORS 限制可信来源
- JWT 密钥启动时安全检查

### 3.5 可扩展性

- 禁用词库可动态更新，无需重启服务
- 支持新增市场/类别的规则扩展
- 检测引擎可插拔，支持新增检测规则（BaseChecker 抽象基类）
- 电商平台对接可扩展（BasePlatformClient 抽象基类）

## 4. 数据需求

### 4.1 禁用词库

- 按市场+类别+违规类型组织
- 支持中英文双语
- 支持正则表达式模式匹配（英文使用 `\b` 词边界）
- 可通过管理界面增删改

### 4.2 法规数据

- 法规名称、编号、条款
- 法规原文摘要
- 与违规类型的关联关系

### 4.3 违规案例

- 真实违规案例及处罚结果
- 按市场+类别分类
- 用于辅助判断和用户参考

### 4.4 替换映射

- 违规词汇 → 合规表述的映射表
- 按市场+类别+违规类型组织
- 上下文感知：仅替换被标记为违规的词汇

### 4.5 用户反馈数据

- 反馈类型（误报/漏报/正确）
- 关联的违规项和检测报告
- 用于驱动规则优化

## 5. 接口需求

### 5.1 RESTful API

- `POST /api/v1/check` — 提交合规检测（需认证）
- `POST /api/v1/check/batch` — 批量合规检测（需认证）
- `POST /api/v1/check/image` — 图片合规检测（需认证）
- `GET /api/v1/reports` — 获取检测报告列表
- `GET /api/v1/reports/{id}` — 获取报告详情
- `DELETE /api/v1/reports/{id}` — 删除报告（需管理员）
- `GET /api/v1/reports/{id}/export/pdf` — 导出 PDF（需认证）
- `GET /api/v1/labels` — 查询必需标签
- `GET /api/v1/certifications` — 查询必需认证
- `GET /api/v1/markets` — 获取支持的市场列表
- `GET /api/v1/categories/{market}` — 获取市场支持的产品类别
- `POST /api/v1/auth/login` — 用户登录
- `POST /api/v1/auth/register` — 用户注册
- `GET /api/v1/auth/me` — 获取当前用户信息
- `POST /api/v1/feedback` — 提交检测反馈（需认证）
- `GET /api/v1/feedback/accuracy` — 检测精度指标（需认证）
- `GET /api/v1/feedback/suggestions` — 规则优化建议（需认证）
- `GET /api/v1/platforms` — 平台连接状态（需认证）
- `POST /api/v1/patrol` — 触发合规巡检（需管理员）
- `GET /api/v1/patrol/history` — 巡检历史（需认证）

### 5.2 请求/响应格式

所有接口使用 JSON 格式，遵循 OpenAPI 3.0 规范。

## 6. 约束与假设

### 约束

- 图片检测依赖 Tesseract OCR，识别精度受图片质量影响
- AI 语义检测依赖外部 LLM API，需配置 API Key
- 电商平台对接需配置各平台 API 凭据
- 默认管理员密码首次部署后应立即修改

### 假设

- 用户提交的产品描述为纯文本，已去除 HTML 标签
- 目标市场法规数据由平台维护，用户不可自行修改
- LLM API 返回的 JSON 格式稳定，有容错解析
